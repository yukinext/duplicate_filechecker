# Implementation Plan

- [ ] 1. Foundation: Typerサブコマンド再編の前提を整える
- [ ] 1.1 コマンド境界と実行契約を整理する
  - 重複チェック系とメンテナンス系の責務境界を `check` / `maint` で明確化する。
  - 既存の重複チェック処理が `check` へ移行しても挙動互換を維持する前提を定義する。
  - 実装開始時点でCLI再編の対象範囲と非対象範囲がチーム内で一貫している状態にする。
  - _Requirements: 5.1, 5.2, 5.3_

- [ ] 2. Core: メンテナンス処理の基盤コンポーネントを実装する
- [ ] 2.1 (P) DBエントリー列挙と削除操作を提供する
  - DBから全エントリー（パス、ハッシュ）を取得できるようにする。
  - 不存在パスの削除に必要な単一エントリー削除操作を提供する。
  - メンテナンスサービスから取得と削除を一貫して呼び出せる状態にする。
  - _Requirements: 1.1, 2.1_
  - _Boundary: Database_

- [ ] 2.2 (P) 削除監査ログの追記出力を提供する
  - `logs/purged_entry.csv` への追記出力を実装する。
  - 出力をタブ区切り3列（ファイルパス、ハッシュ値、UTC ISO 8601時刻）に固定する。
  - `logs` が存在しない環境でも監査ファイルが生成される状態にする。
  - _Requirements: 3.1, 3.2, 3.3_
  - _Boundary: PurgeAuditWriter_

- [ ] 2.3 メンテナンスサービスで判定・削除・継続制御を実装する
  - 全エントリーを走査し、存在しないパスのみを削除対象にする。
  - 削除成功時だけ監査出力を行い、存在するエントリーは保持する。
  - エントリー単位で例外をログ出力し、他エントリー処理を継続する。
  - 処理結果として scanned/purged/failed を返す状態にする。
  - _Depends: 2.1, 2.2_
  - _Requirements: 1.1, 2.1, 2.2, 4.1, 4.2_

- [ ] 3. Integration: Typerコマンド体系を `check` / `maint` へ統合する
- [ ] 3.1 `check` サブコマンドへ重複チェック機能を移行する
  - 既存重複チェック機能をTyperの `check` で起動できるようにする。
  - 既存引数（探索ディレクトリ、パターン、merge関連）が継続して利用できるようにする。
  - `check` 実行で従来の重複チェック処理が完了する状態にする。
  - _Requirements: 5.1_

- [ ] 3.2 `maint` サブコマンドへメンテナンス機能を統合する
  - メンテナンス処理をTyperの `maint` 配下から起動できるようにする。
  - `maint` 実行でサービス層を呼び出し、処理サマリーを出力する。
  - `maint` 実行でDB確認・不要エントリー削除・監査出力まで完了する状態にする。
  - _Depends: 2.3_
  - _Requirements: 1.2, 5.2_

- [ ] 3.3 エントリポイントをTyperルーティングへ一本化する
  - `main.py` の独自ディスパッチを除去し、Typer起動のみに責務を限定する。
  - 手動分岐がなくても `check` と `maint` の両系統が起動できるようにする。
  - エントリポイントが将来のサブコマンド追加時も二重ルーティングを持たない状態にする。
  - _Depends: 3.1, 3.2_
  - _Requirements: 5.3_

- [ ] 4. Validation: 単体・統合テストで要件適合を固定する
- [ ] 4.1 メンテナンス処理の単体テストを整備する
  - DB列挙/削除、存在判定、監査出力フォーマットを検証する。
  - 例外時のログ出力と継続処理（failedカウント）を検証する。
  - 正常系で削除対象のみが削除・監査される状態を確認できるようにする。
  - _Requirements: 1.1, 2.1, 2.2, 3.1, 3.2, 3.3, 4.1, 4.2_

- [ ] 4.2 CLI統合テストを更新する
  - `check` と `maint` のサブコマンド起動を検証する。
  - エントリポイントが手動分岐なしでTyper経由に統一されていることを検証する。
  - コマンド再編後も既存重複チェックの主要フローが回帰していない状態を確認する。
  - _Requirements: 1.2, 5.1, 5.2, 5.3_

- [ ] 5. Regression: 全体回帰検証を完了する
- [ ] 5.1 既存機能と新コマンド体系の回帰確認を実施する
  - 重複チェックとメンテナンス処理をそれぞれ実行し、双方の期待ログと集計結果を確認する。
  - 新コマンド体系で既存テスト群が通過することを確認する。
  - 回帰検証完了後に未解決の障害が残っていない状態にする。
  - _Requirements: 4.1, 4.2, 5.1, 5.2, 5.3_
