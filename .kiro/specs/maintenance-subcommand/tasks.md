# Implementation Plan

- [ ] 1. Foundation: メンテナンス機能の実行基盤を整える
- [ ] 1.1 メンテナンス処理の共通入出力モデルを定義する
  - メンテナンス処理で利用するエントリー情報と実行サマリーの扱いを統一する。
  - 例外発生時でも継続実行できる集計方針（scanned, purged, failed）を明確化する。
  - 実装側で同一モデルを利用して処理結果が一貫して集計される状態にする。
  - _Requirements: 4.2_

- [ ] 2. Core: DB走査・削除・監査出力を実装する
- [ ] 2.1 (P) DBエントリーの全件取得と単一削除を提供する
  - DBからファイルパスとハッシュ値の全エントリーを取得できるようにする。
  - ファイルパスをキーに対象エントリーのみを削除できるようにする。
  - メンテナンス処理からDB走査と削除が呼び出せる状態にする。
  - _Requirements: 1.1, 2.1_
  - _Boundary: DatabaseExtension_

- [ ] 2.2 (P) 削除監査のTSV追記出力を実装する
  - `logs/purged_entry.csv` へタブ区切り3列で追記する出力処理を用意する。
  - 出力列は「ファイルパス」「ハッシュ値」「UTC ISO 8601時刻」に固定する。
  - `logs` ディレクトリがない場合は自動作成され、削除成功時に1行追記される状態にする。
  - _Requirements: 3.1, 3.2, 3.3_
  - _Boundary: PurgeAuditWriter_

- [ ] 2.3 メンテナンスサービスで存在判定・削除・継続制御を実装する
  - 取得した全エントリーを順次走査し、ファイル存在判定を行う。
  - ファイル不存在時のみ削除と監査出力を行い、存在時は保持する。
  - エントリー処理単位で例外を捕捉し、対象パスと例外内容をログ出力して処理継続する。
  - 実行後に走査件数・削除件数・失敗件数を返せる状態にする。
  - _Depends: 2.1, 2.2_
  - _Requirements: 2.1, 2.2, 4.1, 4.2_

- [ ] 3. Integration: CLIサブコマンドとして公開する
- [ ] 3.1 メンテナンス用サブコマンドを既存コマンドと分離して追加する
  - 既存重複チェックコマンドとは独立したサブコマンドとして起動できるようにする。
  - メンテナンスサービスを呼び出し、終了サマリーをログ/コンソールへ出力する。
  - サブコマンド実行でDB全件走査から不要エントリー削除まで一連処理が完了する状態にする。
  - _Depends: 2.3_
  - _Requirements: 1.2, 1.1_

- [ ] 4. Validation: 回帰防止テストを整備する
- [ ] 4.1 DB処理と監査出力の単体テストを追加する
  - 全件取得・削除のDB動作を検証する。
  - 削除監査出力の列順、タブ区切り、UTC ISO 8601形式を検証する。
  - 正常系テストで削除対象のみ監査行が追加されることを確認できる状態にする。
  - _Requirements: 1.1, 2.1, 2.2, 3.1, 3.2, 3.3_

- [ ] 4.2 サブコマンド統合テストと例外継続テストを追加する
  - サブコマンドが既存コマンドと独立して実行できることを検証する。
  - 途中で例外が発生しても残りエントリー処理が継続することを検証する。
  - 例外時のログ出力内容（対象パスと例外）を検証し、処理結果サマリーが返る状態を確認する。
  - _Requirements: 1.2, 4.1, 4.2_
